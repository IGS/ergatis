Use case Run PROmer on all assemblies contained in a BSML repository in an "all against all" manner
Description Generate PROmer search data from a data repository. The data is generated in a "all vs. all" manner by searching all sequences in the repository against each other.
Actors user,developer
Assumptions/Pre-conditions

The BSML_repository and FASTA_repository for the project must be generated via chado2bsml.
The repository must be in a "valid" state as reported by check_repository.
The project repository must have sufficient space available to store results.
The user must have permission to write to the project area.
Post-condition BSML encodings of the promer results are stored in the BSML_repository
Steps
  1. Check repository status via check_repository
  2. Execute run_promer -d [project_name]
  3. Execute workflow monitor, run_wf_monitor -i [instance file], using the instance filename reported by Step 2.
  4. REPEAT
    1. Monitor workflow status
      UNTIL workflow completed
  5. Execute check_repository
Example invocations /usr/local/devel/ANNOTATION/shared/bin/run_promer -d tryp
Variations (optional) If the execution is not successfull, execute 'rerun_wf -i [instance file]' and continue with Step 4

Use case Run PROmer on a set of assemblies against all assemblies in the BSML repository
Description Generate PROmer search data from a data repository. A single sequence is searched all sequences in the repository.
Actors user,developer
Assumptions/Pre-conditions The BSML_repository and FASTA_repository for the project must be generated via chado2bsml.
The repository must be in a "valid" state as reported by check_repository.
The input sequence identifier must be a valid sequence identifier for the project corresponding to a genomic sequence record stored in the data repository.
The project repository must have sufficient space available to store results.
The user must have permission to write to the project area.
Post-condition BSML encodings of the PROmer results are stored in the BSML_repository
Steps
  1. Check repository status via the check_repository utility
  2. Execute run_promer -d [project_name] -a [sequence_identifier]
  3. Execute workflow monitor, run_wf_monitor -i [instance file], using the instance filename reported by Step 2.
  4. REPEAT
    1. Monitor workflow status
      UNTIL workflow completed
  5. Check repository status via the check_repository utility
Example invocations /usr/local/devel/ANNOTATION/shared/bin/run_promer -d tryp -a lma2_100_assembly
Variations (optional) If the execution is not successfull, execute 'rerun_wf -i [instance file]' and continue with Step 4

Use case Run PROmer on all assemblies contained in a BSML repository in an "all against all" manner using a custom configuration.
Description Generate PROmer search data from a data repository. The data is generated in a "all vs. all" manner by searching all sequences in the repository against each other.
Actors user,developer
Assumptions/Pre-conditions The BSML_repository and FASTA_repository for the project must be generated via chado2bsml.
The repository must be in a "valid" state as reported by check_repository.
The project repository must have sufficient space available to store results.
The user must have permission to write to the project area.
Post-condition BSML encodings of the PROmer results are stored in the BSML_repository
Steps
  1. Check repository status via the check_repository utility
  2. Execute run_promer --printconf > myconf to print the configuration setting to a file named myconf
  3. Edit myconf to set configuration settings
  4. Execute run_promer -d [project_name] -c [myconf]
  5. Execute workflow monitor, run_wf_monitor -i [instance file], using the instance filename reported by Step 2.
  6. REPEAT
    1. Monitor workflow status
      UNTIL workflow completed
  7. Check repository status via the check_repository utility
Example invocations /usr/local/devel/ANNOTATION/shared/bin/run_promer -d tryp -c myconf
Variations (optional) If the execution is not successfull, execute 'rerun_wf -i [instance file]' and continue with Step 4
The project_name can also be specified in the config file. run_promer can then be invoked without the -d parameter: run_promer -c myfile.
